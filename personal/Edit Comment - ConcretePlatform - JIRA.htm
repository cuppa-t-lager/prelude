        Description

        Summary:
        A user can save files to their 'My Folders' even if they cannot download them. Visibility of the files is all that's needed, and at least at first, once these files are in their My Folders area they continue to be view only, and cannot be downloaded.

        However, it seems that if you have some files in a My Folder that are view only (download button not present for any of them) you can then enter and exit the File Details screen for one of the files and you will find the Download button is now present (and functional) for all files in your My Folder.

        This is obviously quite an easy way around a download restriction once known.

        Steps to Reproduce:

        1. Log into a client and nominate a target normal user and record their organisation.
        2. Set up some files so that they are restricted so that user's organisation can view them but not download (ensure the user's Zone privilege is suitable too).
        3. Log in as your target user and verify you cannot download the files in their folder location.
        4. Select the files and then use the Actions drop-down in the Workspace to save them to a My Folder folder.
        5. Navigate to your My Folder containing these files and confirm you still cannot download them (it should have defaulted to List View, meaning the download icon would normally display on the very right-hand side of that file's entry in the list).
        6. Now enter a file to view its File Details screen (shouldn't be able to download here either) and then return to the My Folder containing the files via the breadcrumb link. The bug does not reproduce if using the back button on your browser.
        7. Finally, try again to download them from here - the download button for each file should now be visible on the right-hand side of each file's entry on the list.

          con.platform.myFolders.allowShareViewDownload
          My Folders Share View Download Right
          This right allows overriding of a users permissions to share, view and download assets from myFolders

          For now its agreed that this is the correct behaviour as were presuming (PS to confirm) that this is a particular setting for users to override the default settings

          Bob Ferry added a comment - Yesterday
          TESTS COMPLETED: FAILED

          Summary:
          After testing this, it's clear that the core changes this ticket intended have worked; Users cannot now download files in My Folders unless they would normally have permission to do so.

          However, in the course of testing this I have found several problems with storing files in, and downloading files from, My Folders. It looks like all but one of these occurs on Master branch, and so is a bug in Production already. But one of the issues I found does not seem to occur on Master branch, and even though its possible it is not connected with this work it does seem to relate to permissions, and so I want a dev to take a look at it before it is signed off.

          All the issues are listed below. The first is the issue that needs looking at and does not occur on Master. The others are all issues observed on Master which I plan to enter separate tickets for, or a one-off bug I could not reproduce. Of these the first is the most serious. The last issue is also worth mention because it was a bug I saw ONCE only, or rather it occurred for several files during a short period of time, but then I could not get it to occur again. Details below.

          • For some users, mp4 files cannot be previewed in File Details. A determining factor in this is whether a user has the rights to download the file. This does not seem to occur on Master, and its likely it might also block previews of other file types:
            • If I setup Folder A, containing an mp4, so that User A is not allowed to download from Folder A, but give User A Admin Zone privileges, then they will be able to enter Folder A, view that mp4 and watch it/preview it, then download it.
            • If I then remove User A's Admin Zone privileges and go back to Folder A as User A, I will no longer be able to preview the mp4 if I enter its File Details page.
            • Likewise if Folder B was setup which had no file restrictions, User A would be able to enter it and preview an mp4 there too.
          • Users can be redirected to a My Folders page for a file even though they clicked on the file in Share/its original location. Some circumstances mean that a user ends up seeing a File Details page for a file which has a My Folders breadcrumb, and this can occur even when the file you clicked on is not actually in My Folders. The fact that the file doesn't have to exist in My Folders (but some of your files do) can cause some odd behaviours. But this occurs in Master, and thus is likely to be occurring in Production. The repro is probably unlikely to occur for many users, and is a byproduct of looping over the same processes for the sake of testing, but it might be possible that our latest changes have slightly increased the likelihood of this occurring it's just difficult to tell because the steps to reproduce it are hard to pin down. More detail will follow in a ticket.
          • Less serious, and also occurs on Master, is the fact that Admin zone privileges do not trump download restrictions when viewing files in My Folders folders. This occurs in Production but might have been made more obvious by virtue of fixing this loophole for downloads.
          • This is the bug which occurred ONCE only and I could not reproduce. Clicking to download some files resulted in user being directed to the Publish Home Page. This occurred for me several times in a row, and then did not occur when trying to reproduce the bug later (and therefore also could not verify if it occurred on Master). It only occurred when trying to single-click download files that were restricted by folder permissions but allowed a download on the basis of Admin Zone privileges. A user clicking to download those was sent to the Publish Homepage instead. Will keep an eye open for it.

          In order to sign this off today so I can talk to John tomorrow about next steps I haven't yet entered tickets for any of the above. Will aim to do so tomorrow. But Andrzej Michalski and John Nicola might both want to review these issues and I'll be available to clarify details (as I know these repro steps and details are a bit rushed).

          Assigned to John Nicola for further action.
